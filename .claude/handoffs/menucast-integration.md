# menu-cast ↔ content-engine integration — "bake once, mark live"

## Goal

content-engine **fully replaces** menu-cast's (ScreenFire's) screen-generation brain.
menu-cast keeps only: Toast POS sync → AI curation → the clean, filtered menu. That menu is
handed to content-engine, which bakes the finished HTML screens that Fire TVs display.
Done = a restaurant's screens are generated by content-engine end-to-end, with live stock
changes handled without re-running the LLM pipeline.

**How to use this log (rule revised 2026-07-12 by Kal):** this log is **ADVISORY ONLY** —
a record of what was considered and why, not constraints. The decisions were made against a
2026-07-02 snapshot of both codebases and both have moved a lot since (engine: D18 → D77;
composition painting made generation much cheaper, undermining the "generation is slow and
expensive, cache aggressively" premise). Re-derive the design from today's code and
re-litigate freely; keep a past decision only where it still makes sense on its own merits.
The fixed product intent (I1/I2 — full replacement, engine emits finished screens) stands.

**Kal's revision pass (2026-07-12):** I4 invalidated (no upfront daypart pre-baking) ·
I3, I5 rethink · I7 pending cost/size math · I9 never confirmed.

## Plan / Approach

> ⚠ 2026-07-12: this three-tier plan was designed when a bake was assumed slow/expensive.
> Tier 1 stands; tiers 2–3 and the caching around them are under rethink (see I3–I5, I7).

Three-tier response to menu change (cheap → expensive):

1. **Item 86'd** → strike in place at serve time. No reflow, no LLM.
2. **Whole category empties** → strike the whole block immediately (wall never lies), AND
   kick off a background repaint of just that screen with the dead category dropped; swap
   in when it lands.
3. **Daypart boundary** → instant swap to a pre-baked variant (all variants baked upfront).

Pipeline shape: menu-cast enqueues bake tasks → Cloud Run worker runs content-engine →
writes HTML to R2 + variant index to Firestore → touches `lastUpdated` → TVs reload.
Serving stays at menu-cast's `/d/[token]`: fetch baked HTML from R2 → serve-time strike
transform → serve.

## Key Decisions

**I1. Full replacement at the clean-menu seam.**
menu-cast deletes its whole generation stack: `lib/menu/{design,layout,measure,packer,compiler}`,
`lib/rendering/*`, the design/layout/compiler workflow steps, and the render half of
`app/d/[token]/route.ts`. Upstream (pos/curation/live-generator) and downstream (Firestore,
R2, HMAC display tokens, `/d` shell, admin UI) stay.
*Grounded on (2026-07-02 scan):* the seam is `StoredCategory[]` from
`filterStoredCategories`/`filterSnapshot` (`lib/rendering/live-filter.ts:183/224`);
current architecture is stored arrangement-only `LayoutSpec` + request-time `resolveFlow`;
nothing is baked today; no admin approval gate exists.
*Re-verify:* menu-cast file layout & seam types before implementation.

**I2. Seam output = baked, finished HTML screens.**
Chosen over (a) engine emits arrangement-spec-only keeping ScreenFire's renderer, and
(b) slotted-HTML shell with client hydration. Consequence: ScreenFire's request-time
"re-join spec against live data" freshness model is abandoned; freshness is handled by I3–I6.

**I3. Change-response ladder (see Plan). — ⚠ RETHINK (2026-07-12).**
Single 86 = strike in place — layout is frozen, fit is sacred (screens are packed no-scroll;
reflow would break the engine's careful fit). Category-empty = strike all + background
single-screen repaint. Recovery (items back in stock) = swap back to the original bake and
un-strike — **no repaint needed**.
*Grounded on:* ScreenFire's 5-min inventory sync cadence; engine packs fixed-fit screens.
*Rethink because:* cheap generation may collapse the upper tiers into "just regenerate";
strike-in-place (tier 1) likely survives as the instant response while a regen runs.

**I4. ~~Dayparts: pre-bake ALL variants at generation time~~ — ✗ INVALIDATED (2026-07-12, Kal).**
Was: bake every daypart composition upfront, swap on a clock (chosen over lazy+pre-warm).
No longer valid — generation is now cheap enough that on-demand / lazy generation is back
on the table. Replacement approach to be brainstormed with real numbers (lazy+pre-warm B′
is the natural candidate, or even pure on-demand near the boundary).

**I5. Bakes are immutable cache entries. — ⚠ RETHINK (2026-07-12).**
Key: `(restaurant, screen, daypart, menuVersion)`. Repaints (from category-collapse) are
*additional* entries keyed by live composition, never overwrites. Serve route picks:
repaint matching current live composition if present, else the daypart bake.
*Rethink because:* tied to I3/I4 — if the answer to change is usually "regenerate", the
swap-back/variant-picking machinery may shrink to a much simpler current-screen + history.

**I6. Strike mechanism = serve-time transform** (rejected: client-side live-marking JS).
Requirements it imposes on content-engine: (a) every painted item row carries
`data-item-guid` — enforce via the existing binding-integrity QA check; (b) themes ship
`.sold-out` styling in CSS so the transform is literally "add a class". menu-cast's `/d`
route applies classes for dead guids and the existing `lastUpdated` Firestore touch makes
TVs reload. The TV stays a dumb locked browser.

**I7. Storage split: HTML → R2, metadata → Firestore. — ⚠ PENDING MATH (2026-07-12).**
Was forced by: engine output with data-URI-inlined images is multi-MB, over Firestore's
1 MiB doc cap. Firestore keeps the variant index, the **category→screen link** (needed to
route an 86 to its screen), and inventory state. R2 already serves ScreenFire's images.
*Kal's take:* probably unnecessary now that generation is cheap — but **do the math before
deciding**: (a) per-screen generation cost & latency, composed vs free-paint; (b) actual
artifact size today (data-URI inlining may still force a blob store on size alone, no
matter how cheap generation gets — cost and size are independent constraints).

**I8. Bakes run in a separate Cloud Run worker** (chosen over in-process Next.js; over
defer-and-run-locally). Precedent: ScreenFire's `toast-etl` is already a separate Cloud Run
job. A bake is N screens × dayparts × paint→critic loops — ~10+ min, Chromium + LLM heavy;
isolate it from the serving path. Worker: Cloud Run concurrency=1, maxInstances = parallel-bake
budget.

**I9. Dispatch via Cloud Tasks — PROPOSED, NOT YET CONFIRMED by Kal.**
Over Pub/Sub (push-ack ceiling ~10 min is awkward for long jobs), over direct Cloud Run Jobs
API (loses dedup/retry), over a self-hosted broker (ops weight at a handful of restaurants).
Named tasks `bake-{rid}-{menuVersion}` for dedup (≤24 h window). **Per-(screen,daypart)
task granularity** — fits the 30-min `dispatchDeadline` ceiling, gives parallel painting and
per-variant retry; a finalize step/counter flips the variant index when all variants land.
*Quotas verified 2026-07-05:* 500 dispatch/s/queue, `maxConcurrentDispatches` ≤ 5,000,
1 MiB payload, retention 31 d. Limits are irrelevant at our scale; LLM budget binds first.
*Note (2026-07-12):* the per-(screen,daypart) granularity was inherited from I4; with I4
invalidated, granularity is per-screen (or whatever replaces I4). Queue choice itself is
also softer now — if a bake drops to seconds-cheap, even direct dispatch may suffice.

**Superseded early advice (do not follow):** before the full-replacement pivot, the plan was
"menu-cast consumes the engine's *pure core* and injects its own AI-SDK/puppeteer ports to
avoid double stacks in the Next.js Lambda." The worker decision (I8) voids that reasoning —
the worker container can simply use `content-engine/node` (OpenRouter + Playwright) as-is.
Final call parked under Open Questions.
